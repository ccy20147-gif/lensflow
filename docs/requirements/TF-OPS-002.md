# 成本、配额、预算与用量

## 1. 元数据

- ID：TF-OPS-002
- 标题：成本、配额、预算与用量
- 状态：defined
- 目标版本：V0 -> V1 Core
- 优先级：P0
- 全局位置：设置/运行内核
- 直接依赖：TF-PLT-001、TF-WF-003、TF-WF-006、TF-OPS-001
- 责任域：FinOps/运行时
- 个人 DRI：待指派

## 2. 背景与问题

生成式媒体调用成本高且可能因重试、fallback、异步失败和 Provider 计费延迟产生偏差。用户需要在运行前看到估算，在运行后看到实际用量；平台需要在超预算前阻断而不是事后告知。

V1 可以没有支付系统，但不能没有可审计账本和限额。

## 3. 目标与非目标

目标：

- V0 提供运行前估算、基础预算阻断和实际成本展示。
- V1 Core 提供 owner/project/run/node 配额、预留、结算和对账。
- 将每项费用关联到 ProviderInvocationRecord 和 Attempt。
- 正确处理取消、重试、fallback、退款和迟到费用。

非目标：

- 不实现支付、充值、发票或收益分成。
- 不承诺估算等于最终账单。
- 不允许客户端自行声明实际成本。

## 4. 用户与权限

- 项目 owner 可以查看用量并设置不超过账户上限的项目预算。
- 平台运营配置 Provider 价目与全局保护上限。
- 编译器读取预算与配额摘要，不读取账单敏感明细。
- Worker 只能提交绑定 InvocationRecord 的 usage。
- 支持人员按最小权限执行对账与调整，所有调整审计。

## 5. 用户场景与主流程

1. 编译器根据计划、Provider policy 和数量估算成本范围。
2. 用户确认运行，系统检查配额并预留预算。
3. 每个 Provider 调用完成后记录 usage 和实际成本。
4. fallback、retry 和取消分别追加账目，不覆盖原项。
5. Run 结束后释放未使用预留并生成估算偏差。
6. 延迟账单或退款通过对账条目调整可用额度。

## 6. 功能需求

- FR-1：价目必须按 Provider、模型、计费单位、币种和生效时间版本化。
- FR-2：编译必须输出低/中/高或明确算法的成本估算及假设。
- FR-3：运行启动前必须检查 owner、project 和 run 预算。
- FR-4：预算预留必须原子且幂等，避免并发超卖。
- FR-5：实际用量必须来源于 ProviderInvocationRecord 或受审计人工调整。
- FR-6：每条账目关联 run、node、attempt、provider invocation 和 actor。
- FR-7：retry、fallback 和重复生成分别计费并展示原因。
- FR-8：取消时释放未消费预留，已产生 Provider 费用仍记实际账。
- FR-9：重复回调或 usage 上报不得重复记账。
- FR-10：估算与实际偏差超过阈值时产生诊断。
- FR-11：超预算必须在编译或调度前阻断，并给出安全可行动信息。
- FR-12：V1 Core 支持日/月配额和并发运行额度，但不要求支付。

## 7. 交互与展示

- 运行确认展示估算范围、主要假设和预算剩余。
- 运行页展示总成本、节点成本、失败/重试/fallback 成本。
- 设置页展示项目预算、用量趋势和触发的阻断。
- 金额同时显示币种和计费单位，避免伪精度。
- 成本调整显示来源、原因和操作者。

## 8. 数据、类型与公共接口

核心记录包括 PriceRevision、BudgetPolicyRevision、QuotaCounter、CostReservation、UsageLedgerEntry 和 ReconciliationAdjustment。

UsageLedgerEntry 引用 ProviderInvocationRecord，不复制或修改其实际调用事实。

估算进入 CompiledExecutionPlan；实际成本进入运行汇总和账本。金额使用定点小数与明确币种。

## 9. 状态机与业务规则

ReservationStatus 为 pending、held、partially_consumed、released、expired。

账本采用追加记录；退款和更正使用反向或 adjustment 条目，不覆盖原费用。

并发预留使用数据库原子条件更新。Run 终结后由结算任务释放剩余预留。

## 10. 失败、降级与恢复

- 价目缺失时不能给出伪精确估算；按策略阻断或明确未知上限。
- Provider 未返回 usage 时标记 pending_reconciliation，并使用可解释临时估算。
- 结算任务失败可幂等重跑。
- 取消与回调竞争时以实际 InvocationRecord 和追加账本收敛。
- 对账服务不可用不删除调用记录，预算采用保守可用余额。

## 11. 安全、隐私、内容与授权

- 用户只能查看其 owner_scope 内用量。
- 客户端提交的成本仅作显示请求，不成为账本事实。
- 价格合同、供应商折扣和平台级密钥按敏感信息限制。
- 手工调整要求双重权限或事后复核，并记录理由。
- 用量导出不得包含 prompt、素材或 secret。

## 12. 观测与运营

- 监控估算偏差、未对账调用、预留泄漏、超预算阻断和重复账目。
- 按 Provider/模型跟踪单位成本、失败成本和 fallback 成本。
- 对异常用量增长、负余额和长时间 held 预留告警。
- 生成每日账本完整性与 Provider 对账报告。

## 13. 验收标准

- AC-1：V0 真实运行前展示估算，完成后显示与 InvocationRecord 一致的实际成本。
- AC-2：两个并发运行竞争同一剩余预算时，不得同时超额预留成功。
- AC-3：重复 usage 回调十次只生成一条有效费用。
- AC-4：取消运行后未用预留释放，取消前已计费请求仍保留。
- AC-5：fallback 与 retry 的额外成本在节点与 run 汇总中可单独识别。
- AC-6：人工调整不覆盖原账目，并可追溯 actor、原因和复核。

## 14. 测试场景

- 正常：估算、预留、调用记账、结算和释放。
- 边界：零成本、未知价目、多币种输入、最大预算和临界余额。
- 失败：usage 缺失、对账中断、重复回调、负向调整和 Provider 延迟账单。
- 权限：跨 owner 用量、客户端伪造成本和无权调整。
- 并发/恢复：预留竞争、取消/回调竞争和结算任务重跑。

## 15. 交付与回退

- V0 交付单币种基础价目、估算、预算和实际成本。
- V1 Core 交付多级配额、完整账本、对账和趋势。
- 新价格 revision 先验证，不重算历史账目。
- 回退新结算器时保留追加账本，由旧版本处理其支持的记录。

## 16. 已决策事项与开放问题

已决策：运行前估算/预留，运行后按实际调用记账；V1 不依赖支付系统。

开放问题：产品首发币种与预算默认值由商业配置决定，但所有数值必须版本化并可审计。
