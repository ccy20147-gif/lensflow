# 需求真相源与变更控制

## 1. 元数据

- ID：TF-GOV-001
- 标题：需求真相源与变更控制
- 状态：in_delivery
- 目标版本：Foundation
- 优先级：P0
- 全局位置：全局治理
- 直接依赖：无
- 责任域：产品治理
- 个人 DRI：main-agent

## 2. 背景与问题

LensFlow 同时包含工作流运行时、创作工作台、Agent、媒体生成和社区授权。若需求只存在于会话、旧蓝图或实现代码中，同一概念会出现不同版本，发布也无法证明是否完成。

旧蓝图和参考仓库只能作为证据。正式范围、冻结决策、依赖、版本和验收必须由主需求表及其详细需求文档给出。

## 3. 目标与非目标

目标：

- 为每项正式需求提供唯一 ID、版本、状态、责任域、依赖和验收证据。
- 让范围变化先完成影响分析，再进入设计和实现。
- 形成从 Master ID 到发布监控的完整证据链。

非目标：

- 不替代研发排期、缺陷管理或人员绩效工具。
- 不把探索性研究自动升级为产品承诺。
- 不允许详细需求自行推翻主表冻结决策。

## 4. 用户与权限

- 主需求表维护者可以批准需求新增、拆分、延期和 supersede。
- 责任域负责人可以提案并维护所属详细需求，但不能单独改变跨域公共合同。
- 工程、设计、QA 和运营可以提交影响证据与评审意见。
- 普通查看者只读访问已批准记录。
- 所有治理修改必须保留作者、时间、原因和评审人。

## 5. 用户场景与主流程

1. 提案人创建带背景、范围和受影响 ID 的变更请求。
2. 产品治理检查是否触及冻结决策、公共类型、权限、版本门或直接依赖。
3. 责任域完成产品、技术、安全、质量和交付影响分析。
4. 批准后先更新主表，再更新详细需求、ADR、设计和实现计划。
5. 发布验收将实现证据反向关联到 FR、AC 和 Master ID。

## 6. 功能需求

- FR-1：系统必须维护唯一且不可复用的需求 ID。
- FR-2：每个需求必须记录标题、状态、目标版本、优先级、全局位置、直接依赖、责任域和个人 DRI。
- FR-3：每个需求必须关联一份详细需求文档，缺失时不得进入 approved。
- FR-4：变更请求必须记录原因、影响范围、兼容策略、回退方式和裁决。
- FR-5：触及公共合同的变更必须列出全部直接和传递下游需求。
- FR-6：被替代需求必须保留历史内容、替代者 ID 和生效时间。
- FR-7：实现、测试和发布证据必须引用稳定 FR 与 AC 编号。
- FR-8：版本门必须能查询所有 P0/P1 项的状态和阻塞原因。
- FR-9：旧蓝图、研究链接和参考代码只能作为 evidence，不得作为隐式规范。
- FR-10：治理检查必须阻止孤儿详细文档、重复 ID 和未知依赖。

## 7. 交互与展示

- 主表按领域展示需求 ID、版本、状态和详细文档入口。
- 变更记录展示新增、修改、延期、拒绝和 supersede。
- 影响视图展示直接依赖和传递依赖，不用人工搜索全文。
- 发布门视图按版本列出未 approved、未 verified 和缺证据项目。
- 冲突信息必须指出具体 ID、字段和责任域。

## 8. 数据、类型与公共接口

治理记录至少包含 RequirementRecord、ChangeRequest、EvidenceLink 和 ReleaseGateSnapshot。

RequirementRecord 不复制业务 Artifact、Resource、Provider 或运行状态；它只引用对应需求和证据。依赖字段使用正式需求 ID 数组，禁止自由文本代替。

证据链接至少区分 detailed_requirement、ADR、design、schema、implementation、automated_test、E2E、visual_evidence、monitoring 和 rollback。

## 9. 状态机与业务规则

需求状态严格使用：

discovered -> defined -> reviewed -> approved -> in_delivery -> implemented -> verified -> released。

任一阶段可以在有裁决记录时进入 deferred、superseded、rejected。implemented 不等于 verified；released 必须满足版本门、监控和回退要求。

同一变更请求只能由一个最终裁决关闭。并发修改主表时必须基于版本或内容 hash 检测冲突。

## 10. 失败、降级与恢复

- 未知依赖、重复 ID 或缺详细文档时拒绝状态提升。
- 主表与详细需求元数据不一致时，以主表为准并产生阻断项。
- 证据系统不可用时可以保存待同步记录，但不得把需求标为 verified。
- 错误修改通过新提交恢复，不删除历史审计记录。
- 无法自动计算传递依赖时，发布门保持阻断并要求人工裁决。

## 11. 安全、隐私、内容与授权

- 未公开路线、许可意见和安全缺陷证据按最小权限访问。
- 证据链接不得暴露密钥、个人数据或未净化 provider 响应。
- 审计记录只允许追加更正，不允许无痕删除。
- 法律或安全限制的证据可以隐藏正文，但必须保留存在性、责任人和裁决状态。

## 12. 观测与运营

- 记录需求状态变更数、平均评审时长、孤儿证据数和版本门阻塞数。
- 每周检查主表与详细文档元数据漂移。
- 每个发布候选生成不可变 ReleaseGateSnapshot。
- 对未知 ID、失效链接和未裁决 P0 开放问题产生告警。

## 13. 验收标准

- AC-1：给定任一正式需求，查询结果包含第 1 节要求的全部元数据和唯一详细文档。
- AC-2：当变更公共类型时，系统列出所有直接和传递依赖，遗漏一个已登记依赖即失败。
- AC-3：当详细文档状态与主表不一致时，发布门显示阻断且不得标记 released。
- AC-4：任取一个 released 需求，可沿证据链找到 FR/AC、设计、实现、自动测试、E2E 和监控。
- AC-5：删除或改写历史裁决的请求被拒绝，审计日志保留原值和尝试者。

## 14. 测试场景

- 正常：新增需求、评审、实现、验证和发布形成完整记录。
- 边界：需求拆分为两个新 ID，原 ID superseded 且引用关系完整。
- 失败：重复 ID、未知依赖、失效详细文档路径均阻断审批。
- 权限：非维护者不能批准跨域公共合同变化。
- 并发/恢复：两人基于同一版本修改时第二次提交收到冲突；恢复后历史不丢失。

## 15. 交付与回退

- Foundation 交付主表、详细文档模板、变更流程、依赖检查和发布门清单。
- 初期可由版本库文档和 CI 检查实现，不要求自建项目管理产品。
- 回退只撤销有误的新规则或自动化，不删除已形成的需求与裁决历史。
- 发布证据为一次完整变更演练和一次版本门快照。

## 16. 已决策事项与开放问题

已决策：主表是需求真相源；旧蓝图仅作证据；状态提升必须有客观证据。

开放问题：进入 in_delivery 前选择与现有研发工具的同步方式，优先保持单向引用，避免建立第二需求真相源。
