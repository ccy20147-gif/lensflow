# 新 FastAPI 单后端

## 1. 元数据

- ID：TF-ARC-001
- 标题：新 FastAPI 单后端
- 状态：in_delivery
- 目标版本：Foundation
- 优先级：P0
- 全局位置：平台内核
- 直接依赖：TF-GOV-001、TF-GOV-002
- 责任域：平台架构
- 个人 DRI：main-agent

## 2. 背景与问题

Toonflow 与 SeedV demo 包含不同后端、状态和数据假设。直接拼接会产生双运行真相、双权限系统、异步任务冲突和难以恢复的数据链。

产品需要一个全新 FastAPI 后端作为所有业务状态和命令的唯一入口。参考仓库只提供领域与工程证据，不成为生产依赖。

## 3. 目标与非目标

目标：

- 建立可部署、可测试、可观测的新 FastAPI 服务骨架。
- 冻结模块边界、事务边界、API 版本和后台任务接入方式。
- 保证前端、worker 和 provider 回调共享同一权限与状态合同。
- 为后续工作流、资源、社区和运营模块提供稳定宿主。

非目标：

- 不迁移 SeedV/Toonflow 历史数据或运行旧后端。
- Foundation 不实现全部业务功能。
- 不允许浏览器或 LLM 直接写核心状态。

## 4. 用户与权限

- Web 客户端通过受认证 API 使用业务能力。
- Worker 使用服务身份领取任务并以 fencing 条件提交结果。
- Provider 回调通过独立签名入口进入统一命令处理。
- 运维人员访问健康、指标和受控管理接口。
- 模块间调用必须携带 owner_scope、actor 和 correlation ID。

## 5. 用户场景与主流程

1. 服务启动并验证配置、数据库、Blob 和队列依赖。
2. 客户端访问版本化 API，认证中间件建立 actor 与 owner_scope。
3. 路由调用应用服务，不直接操作跨域表。
4. 应用服务在事务中写聚合、审计和 outbox。
5. 异步执行通过持久任务进入 worker，结果回到同一状态机。
6. 客户端通过查询和事件通道读取状态。

## 6. 功能需求

- FR-1：必须创建独立 FastAPI 应用，不导入或启动 Toonflow/SeedV 生产后端。
- FR-2：API 使用明确版本前缀并提供兼容与弃用规则。
- FR-3：模块至少分离身份项目、工作流、资源、运行时、Provider、社区和审计责任。
- FR-4：写操作必须经应用服务校验权限、版本和状态转换。
- FR-5：跨模块一致写入使用事务和 outbox，不依赖内存事件作为真相。
- FR-6：后台 worker、回调和 API 共享相同领域命令及幂等规则。
- FR-7：提供 liveness、readiness、build/version 和依赖诊断端点。
- FR-8：配置和 secret 从环境或受控 secret provider 读取，不写入仓库。
- FR-9：错误响应使用安全错误码和 correlation ID，不返回内部堆栈。
- FR-10：模块边界和禁止依赖由 ADR 与自动检查固定。

## 7. 交互与展示

- 本需求不新增最终用户页面。
- 前端对不可用依赖展示可行动的安全错误与 correlation ID。
- 运维健康页区分进程存活、可接流量和降级依赖。
- API 文档在非生产或受控环境开放，标注认证和版本。

## 8. 数据、类型与公共接口

后端必须原样承载主表第 8 节的 ArtifactVersion、ResourceDraft、ResourceRevision、调用矩阵、Provider 记录和状态族。

所有持久业务表携带 owner_scope 或可由父聚合确定 owner_scope。数据库 schema、迁移和 API schema 由版本库管理。

模块只能通过应用接口或明确只读查询访问其他域；禁止复制 Artifact、Run 或授权真相。

## 9. 状态机与业务规则

核心状态转换由后端领域服务决定。前端、worker、Provider 或 LLM 只能提交命令，不能提交任意目标状态。

命令必须支持幂等键；重复回调返回同一结果或明确 no-op。并发更新使用 revision、epoch 或 compare-and-swap。

服务启动不得自动运行破坏性迁移；迁移先执行兼容检查。

## 10. 失败、降级与恢复

- 数据库不可用时 readiness 失败，服务不接收会写状态的流量。
- 事件投递失败时事务数据保留，outbox 后续重试。
- 非关键 Provider 不可用时核心项目与历史查询仍可访问。
- 重复或乱序回调按幂等键和状态机拒绝。
- 部署失败通过向后兼容数据库和上一镜像回退。

## 11. 安全、隐私、内容与授权

- 默认拒绝未知身份、owner_scope 和跨项目访问。
- 管理接口、健康详情和 API 文档按环境限制。
- 请求体、日志和 trace 进行 secret 与个人数据清理。
- CORS、CSRF、速率限制、上传和回调签名由安全基线统一配置。
- 任何旧 demo 服务不得拥有生产数据库或密钥。

## 12. 观测与运营

- 记录请求率、错误率、延迟、数据库池、任务积压和 outbox 延迟。
- 每个请求和异步任务传播 correlation ID、actor、project 和 run 关联。
- readiness 说明具体失败依赖，但公开响应不泄露凭证。
- 部署记录 build SHA、schema 版本和配置版本。

## 13. 验收标准

- AC-1：从干净环境启动后，liveness、readiness 和版本端点在 60 秒内可用。
- AC-2：代码和运行进程中不存在 Toonflow/SeedV 后端作为必需服务。
- AC-3：一次测试写入同时产生业务记录与 outbox；模拟投递失败后可恢复且不重复业务写入。
- AC-4：跨 owner_scope 请求返回拒绝，且审计记录包含 actor、目标和 correlation ID。
- AC-5：上一兼容版本可在新迁移后回退启动，历史只读数据仍可读取。

## 14. 测试场景

- 正常：认证 API 创建项目并通过查询读回。
- 边界：最大允许请求体、空配置和弃用 API 版本得到明确响应。
- 失败：数据库、Blob 或事件投递分别故障时进入定义的 readiness/降级状态。
- 权限：匿名、跨 owner、伪造 worker 和错误回调签名均被拒绝。
- 并发/恢复：重复命令和乱序回调不产生双写；服务重启后 outbox 继续。

## 15. 交付与回退

- Foundation 交付新服务目录、容器、迁移框架、模块 ADR、健康端点和 CI。
- 部署先进入非生产环境执行 smoke、迁移和回退演练。
- 回退使用上一兼容镜像；不得切回旧 demo 后端。
- 发布证据包括启动日志、健康检查、边界测试和故障演练记录。

## 16. 已决策事项与开放问题

已决策：生产仅有一个新 FastAPI 后端；旧代码库不作为服务或数据库基础。

开放问题：具体关系型数据库、任务队列和部署平台由 Foundation ADR 固定，但选择不得改变本需求的事务、幂等和单真相约束。
