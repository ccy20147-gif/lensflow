# 项目、工作流与资源库外壳

## 1. 元数据

- ID：TF-PLT-002
- 标题：项目、工作流与资源库外壳
- 状态：in_delivery
- 目标版本：V0
- 优先级：P0
- 全局位置：产品外壳
- 直接依赖：TF-PLT-001、TF-WF-004、TF-WF-005、TF-OPS-003
- 责任域：核心产品/前端平台
- 个人 DRI：main-agent

## 2. 背景与问题

创作者需要一个稳定容器组织多个工作流、世界观、角色、镜头、媒体和运行历史。若把项目等同于单个工作流，模板创建、资源复用和后续社区引用都会受限。

V0 需要先交付可恢复的项目外壳，使非技术用户不打开画布也能完成模板流程。

## 3. 目标与非目标

目标：

- 一个项目容纳多个 WorkflowDraft/Revision 和多类 Resource。
- 提供项目创建、切换、保存、资源检索与最近活动恢复。
- 让模板、工作区、画布和运行历史共享项目上下文。
- 对 Blob、Artifact 和 Resource 提供统一入口而不复制真相。

非目标：

- 不实现团队空间、实时协作或社区公开列表。
- 不在前端维护权威运行状态。
- 不把项目删除等同于立即物理删除所有历史证据。

## 4. 用户与权限

- 项目 owner 可以创建、编辑、归档和发起删除。
- V0 bootstrap owner 只能看到自己的项目。
- 系统任务可在限定 scope 内更新运行索引，不能修改用户 Draft。
- 无权用户不能看到项目名称、缩略图或资源数量。

## 5. 用户场景与主流程

1. 用户从模板或空项目入口创建项目。
2. 产品建立项目、默认工作区状态和第一个 WorkflowDraft。
3. 用户上传或生成资源，并在资源库按类型检索。
4. 用户切换工作流或工作区，项目上下文保持一致。
5. 页面刷新后恢复最近项目、位置和后端持久状态。
6. 用户归档或发起删除，系统根据保留策略处理。

## 6. 功能需求

- FR-1：项目必须拥有稳定 project_id、owner_scope、名称、状态和时间戳。
- FR-2：一个项目可以关联零到多个 Workflow 和 Resource，不共享可变真相。
- FR-3：模板创建必须以事务或补偿流程建立完整最小项目。
- FR-4：资源库必须按 resource_type、名称、更新时间和来源检索。
- FR-5：项目首页必须展示最近工作流、资源、运行和失败任务摘要。
- FR-6：前端导航状态可以本地保存，但业务数据必须从后端恢复。
- FR-7：资源条目必须引用固定 ResourceRevision 或明确 Draft 状态。
- FR-8：项目切换必须清空上一项目的敏感缓存与订阅。
- FR-9：归档项目默认只读，恢复后可继续编辑。
- FR-10：删除请求必须调用正式保留与引用保护流程。

## 7. 交互与展示

- 首页直接展示新建、模板、最近项目和继续创作入口。
- 项目内提供工作区、工作流、资源库、运行历史和设置导航。
- 资源库使用类型筛选、搜索、缩略图/列表切换和版本入口。
- 空状态提供明确创建命令，不显示产品功能说明卡片。
- 失败任务和 stale 资源有可定位到节点或工作台的入口。

## 8. 数据、类型与公共接口

Project 记录 project_id、owner_scope、metadata、ProjectStatus 和默认入口。

项目关系只存 Workflow ID、Resource ID、Run ID 等稳定标识；内容读取使用 ArtifactRef、ResourceDraft 或 ResourceRef。

缩略图和 Blob 使用 TF-OPS-003 的私有访问合同。项目外壳不复制 content_json。

## 9. 状态机与业务规则

ProjectStatus 为 active、archived、deletion_pending 和 deleted_tombstone。

active 可归档；archived 可恢复或发起删除。deletion_pending 期间阻止新运行和新上传。

项目创建请求使用 idempotency key；重复模板请求返回同一项目或明确冲突。

## 10. 失败、降级与恢复

- 模板依赖缺失时不创建半成品项目，或执行可审计补偿。
- 缩略图服务不可用时显示类型占位，不暴露 Blob URL。
- 项目切换失败时保持当前项目，不混合两个项目数据。
- 资源索引延迟时允许按稳定 ID 打开已有资源。
- 页面刷新通过 URL project_id 和后端状态恢复。

## 11. 安全、隐私、内容与授权

- 每个项目 API 在后端验证 owner_scope。
- 私有资源缩略图使用短期签名访问并禁止跨项目复用。
- 项目导出、删除和凭证设置要求更高权限。
- 搜索索引不得收录无权内容或未净化敏感文本。

## 12. 观测与运营

- 记录项目创建成功率、模板补偿、切换延迟和资源检索延迟。
- 监控跨项目缓存污染和无权访问拒绝。
- 对 deletion_pending 超时和孤儿 Blob/Resource 产生运营任务。
- 项目恢复事件关联前端版本和 correlation ID。

## 13. 验收标准

- AC-1：同一项目创建两个工作流和三种 Resource 后，刷新与重新登录均可完整检索。
- AC-2：从模板重复提交相同 idempotency key 只创建一个项目。
- AC-3：切换项目后网络请求、事件订阅和缩略图均不再包含上一项目标识。
- AC-4：归档项目不能新建运行，恢复后原 Revision ID 不变。
- AC-5：无权用户访问 project_id 返回安全拒绝且不泄露项目元数据。

## 14. 测试场景

- 正常：新建、模板创建、切换、搜索、归档和恢复。
- 边界：空项目、100 个资源、多个工作流和长名称。
- 失败：模板依赖、Blob 缩略图、索引服务和保存请求失败。
- 权限：跨 owner 项目、签名 URL 重放和删除操作越权。
- 并发/恢复：重复创建、两个标签页改项目名称和刷新后恢复。

## 15. 交付与回退

- V0 交付项目列表、项目壳、工作流入口、资源库和运行摘要。
- 模板入口可按功能开关逐个启用。
- 前端新壳回退时保留后端 Project、Workflow 和 Resource 数据。
- 发布证据包括端到端模板项目和故障恢复录屏/测试报告。

## 16. 已决策事项与开放问题

已决策：项目不是工作流；资源库展示引用，不复制内容真相。

开放问题：V0 是否提供项目封面自动生成不影响 P0，若实现必须保存为普通 ArtifactVersion。
